# 001 — 미국 주식 반등 스크리너 대시보드

`data/stock-universe.md`의 종목들(레버리지 ETF 포함)에 대해 "하락 이후 반등할 확률"을 과거 일봉 데이터로 통계화하여 Next.js 웹페이지 한 장으로 보여준다.

## 핵심 정의 (측정 가능)

- **하락 상태**: 최근 60거래일 고점 대비 종가가 10% 이상 낮은 상태 (drawdown ≤ −10%)
- **반등 사례(episode)**: 과거 시계열에서 drawdown이 −10%를 하향 돌파한 날을 진입일로 보고, 이후 20거래일 안에 종가가 진입일 종가 대비 +5% 이상 회복하면 성공
- **반등 성공률**: 성공 사례 수 ÷ 전체 사례 수 (사례 0건이면 N/A, 5건 미만이면 "표본 부족" 표시)
- **보조 지표**: RSI(14, Wilder)

## 요구사항

R1 (Ubiquitous): 시스템은 항상 종목 유니버스를 data/stock-universe.md 한 곳에서 읽어야 한다
R2 (Event): 사용자가 대시보드에 접속하면, 시스템은 유니버스 전 종목의 반등 통계 표(티커·이름·카테고리·현재 낙폭·RSI·사례 수·반등 성공률·평균 반등폭)를 보여야 한다
R3 (State): 종목이 하락 상태인 동안, 시스템은 그 종목을 "지금 하락 중" 구간에 반등 성공률 내림차순으로 보여야 한다
R4 (Unwanted): 일부 종목의 시세 조회가 실패하는 경우에도, 시스템은 나머지 종목의 표를 렌더하고 실패 종목을 구분 표시해야 한다
R5 (Invariant): 반등 성공률은 입력 시계열과 무관하게 0 이상 1 이하이거나 N/A여야 하고, RSI는 0 이상 100 이하여야 한다
R6 (Ubiquitous): 시스템은 항상 면책 문구(투자 조언이 아님)와 레버리지 ETF 변동성 잠식 경고를 페이지에 표시해야 한다
R7 (Optional): DATA_SOURCE=fixture 환경에서는, 시스템은 네트워크 없이 번들 픽스처 데이터만으로 동작해야 한다
R8 (Ubiquitous): 시스템은 항상 수집한 일봉 시세를 SQLite(data/market.db, 픽스처 환경은 메모리 DB)에 저장하고 통계는 DB에서 읽은 시계열로 계산해야 한다
R9 (State): 종목의 캐시가 신선한(수집 후 24시간 이내) 동안, 시스템은 그 종목의 외부 시세 API를 호출하지 않아야 한다
R10 (Ubiquitous): 시스템은 항상 유니버스를 시가총액 상위 주식 200종목과 운용자산 상위 ETF 100종목(레버리지 포함, data/stock-universe.md 큐레이션)으로 구성해야 한다
R11 (Event): 사용자가 접속하면, 시스템은 요약 카드(유니버스 구성·하락 중 종목 수·하락군 평균 반등 성공률·조회 실패 수·데이터 수집 시각)를 표 상단에 보여야 한다

## 시나리오

S1 (covers: R1,R2,R6): Given 픽스처 환경의 대시보드 / When 루트 페이지에 접속하면 / Then 제목·통계 표·면책 문구·레버리지 경고가 보이고 axe serious 위반이 0건이다
S2 (covers: R3,R5): Given 하락 상태 픽스처 종목 FIXDN(과거 반등 사례 2건 전부 성공) / When 페이지를 열면 / Then FIXDN이 "지금 하락 중" 구간에 성공률 100% 표기로 나타난다
S3 (covers: R4,R7): Given 시세 조회가 실패하도록 만든 픽스처 종목 FIXBAD / When 페이지를 열면 / Then 나머지 종목 표는 정상 렌더되고 FIXBAD는 실패로 구분 표시된다
S4 (covers: R8,R9): Given 메모리 DB로 기동한 픽스처 환경 / When 같은 종목을 두 번 조회하면 / Then 두 번째는 원천 조회 없이 DB에서 응답하고 페이지에 수집 시각이 표시된다
S5 (covers: R10,R11): Given 대시보드 / When 페이지를 열면 / Then 요약 카드 영역에 유니버스 구성·하락 수·실패 수·수집 시각이 나타난다

## [ASSUMED]

1. 시세 출처는 Yahoo Finance v8 chart 엔드포인트(무키, 일봉 2년, adjusted close) — 비공식이므로 종목 단위 실패를 R4로 흡수하고, 페이지는 1시간 ISR 캐시로 호출량을 줄인다.
2. "반등 확률"의 정의는 위 핵심 정의의 백테스트 비율로 한정한다 — 예측 모델이 아니라 과거 빈도 통계다.
3. 파라미터(60일 고점, −10%, 20일, +5%, RSI 14)는 코드 상수 PARAMS로 노출하고 페이지에 명시한다 — 조정은 스펙 개정으로.
4. 인수 테스트는 픽스처 모드(R7)로 결정론을 확보한다 — 실데이터 검증은 수동 운용 확인으로 한정.
5. 대시보드는 루트 페이지(/)에 둔다 — 부트스트랩 자리표시 페이지를 대체한다.
6. (rev2) SQLite 구현은 better-sqlite3(동기, prebuilt 바이너리), 스키마는 prices(ticker,d,close)+fetch_log(ticker,fetched_at,points,error). DB 파일은 런타임 산출물로 gitignore.
7. (rev2) 캐시 TTL 24시간 — 일봉 데이터는 하루 1회 수집이면 충분. ISR(1h)은 렌더 캐시, TTL은 수집 캐시로 역할이 다르다.
8. (rev2) "상위"의 원본 순위 데이터(시가총액·AUM)는 실시간 조회하지 않는다 — 2026-06 시점 큐레이션을 md에 고정하고, 갱신은 md 편집(부정확 티커는 R4가 흡수). 레버리지 여부는 비고의 배율 표기로 판별한다.
9. (rev2) UI는 다크 금융 터미널 톤(IBM Plex Sans KR + IBM Plex Mono, next/font 자체 호스팅), 모션은 CSS 전용, WCAG AA 대비 유지. 클라이언트 JS 추가 없음(서버 컴포넌트 유지).
